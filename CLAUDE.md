Go backend for Pricing Optimizer. This is Git repo #1. When Claude Code is opened here,
this is the active scope (the `.git/` directory stops the upward search into the umbrella).
Cross-cutting product context lives in the umbrella `../.claude/CLAUDE.md`; the full

## What this service does

Analyzes a product URL and generates three pricing-page variations via structured LLM
tool calling, streaming them over Server-Sent Events. See `openapi.yaml` (synced from the
umbrella root) for the contract. The 5 endpoints: analyze, generate (SSE), get generation,
export, healthz.

## Architecture ā€” Clean Architecture, pragmatic four layers

```
cmd/api/            entry point, wiring, graceful shutdown
internal/
  domain/           entities and ports (interfaces); imports nothing
  usecase/          orchestration; imports domain only
  adapter/
    httpapi/        Chi handlers + SSE writer
    scraper/        chromedp (SPA) + colly (static)
    llm/            Anthropic + Groq adapters behind LLMProvider
    repository/     pgx + sqlc
    cache/          Redis client
  api/              generated from openapi.yaml (server.gen.go)
  config/           envconfig struct
  telemetry/        OpenTelemetry setup
db/
  migrations/       goose
  queries/          sqlc
test/
  e2e/              contract tests
  fixtures/         LLM golden responses
```

Layer rules (non-negotiable, enforced in review):
- `usecase` imports `domain` only. Never adapters, never external packages.
- `domain` imports nothing. Pure entities and interfaces.
- `adapter` imports `domain`. Never `usecase`.
- `cmd` imports everything and wires at startup.

## Contract workflow (source of truth)

`openapi.yaml` is generated/owned at the umbrella root, synced here via `make sync-openapi`.
Never edit this repo's `openapi.yaml` directly. Regenerate server code with oapi-codegen
after a sync. Spec is OpenAPI 3.1; see `../docs/decisions/0004-openapi-31-with-oapi-codegen.md`
for the codegen caveat.

## Testing (non-negotiable)

- Table-driven tests with `t.Run` and descriptive English names. Helpers call `t.Helper()`.
- Mocks generated by uber-go/mock, never hand-written.
- Integration tests use testcontainers-go (real Postgres), not mocks.
- LLM adapter tests use mocked clients + golden fixtures in `test/fixtures/`.
- Coverage >= 80% in `internal/usecase` and `internal/domain`.

## Branching workflow (constitution: branch-per-task)

`main` is protected ā€” never develop or commit on it directly. Create the branch BEFORE
writing code, from an up-to-date `main`:
- Spec-driven feature: `NNN-slug`, created by `/speckit-specify` (e.g. `002-generate-variations`).
  Never rename it; `specs/NNN-slug/` must match.
- Standalone change (CI, deps, tooling, docs, hotfix): `<type>/<slug>`, Conventional type +
  kebab-case slug (e.g. `ci/golangci-lint-workflow`).

One branch = one logical unit of work. Before pushing the branch or opening a PR, run the
`pr-reviewer` agent against the local diff (`git diff main...HEAD`) and fix blocking
findings ā€” the PR should already be clean when it opens, not fixed up after with a
follow-up commit. Every change reaches `main` only via a PR that is approved (CI green +
quality gates passing); delete the branch after merge. Use the `start-task` skill to get on
the right branch.

## Project skills (`.claude/skills/`)

- `/start-task` ā€” before any code: ensure clean, up-to-date `main`, then create the
  correctly named branch (routes features to `/speckit-specify`, else `<type>/<slug>`).
- `/go-best-practices` ā€” apply before writing or reviewing Go: layer placement, error
  handling, context, errgroup concurrency, slog, constructors, golangci-lint compliance.
- `/write-go-tests` ā€” write behavior-focused tests: table-driven + testify, uber-go/mock
  generated mocks, testcontainers Postgres, LLM golden fixtures, >=80% usecase/domain.
- `/commit` ā€” craft Conventional Commits with architecture-aligned scopes and no AI/co-author
  attribution; commit only when asked, on a task branch (never `main`).

These are also model-invocable: they load automatically when their task comes up.
